Design Process

  • For project 2, I decided to create a web applciation that gathers data from my spotify account. Specifically from my most recently played songs list and displays them in the form of a chart using Chart.js

  • The process used to design this web application was to create backend scripts to communicate with Spotify's Developer API service. Spotify uses Bearer Oauth token's to validate any developer's web application.

    1. Authorize Web App
    2. Create a script for handling Authorization (key.js)
    3. Another script for data manipulation (music.js)
    4. Attatch inline script for Spotify API (spotify-web-api-js.js)
    5. Attatch another inline script for Chartjs (chart.js)
    6. In key.js specify the redirectc URI for the current project
    7. Now on the web in the Spotify Developer's page, register the application and whitelist the redirect URI
    8. Once page can be redirected, music.js should initialize and create variables necessary to display data.
    9. With the spotify api script attatched, the getMyRecentlyPlayedTracks function can be used.

      (Note: This function uses Promises to control data that has not been fully defined yet.)
    10. Once in the getMyRecentlyPlayedTracks, store the data into a temp variable and loop through all data items.
    11. For each song, traverse the data that is attatched to the data object. Getting only what is necessary.
    12. After getting the necessary data, the song data is then stored in an array and is further used to construct
      the display and chart data.
    13. The track data is then compared to the textfile database holding all songs that were previously listened to.

      (Note: Population of data could not be done in browser due to the CORS policy, population done on localized server)
    14. Track names are compared with the database in the hopes to find the number of times a song is listened to.
      Track names and count are stored in local storage and accessed to help construct the chart data.
    15. Once the data from storage is accessed, the chart can be drawn along with the button slider, playbutton widget,
      expandable legend, and toggable mixed graphs.

Challenges

  1. Due to promise statement in spotify functon, localstorage can only load data after a page refresh. Page refresh detection implemented, performance can vary.

  2. Dynamically creating Playbutton widgets in an iframe with spotify's music links introduces authorization errors. Violating the service workers registration from user denied permission.
    In addition to dom elements being refernced from a different server that the web app cannot account for.

Personal Grade

  • When looking at this project in its totallity, I think it aims to achieve more than what was expected with regards to the kind of Authorization needed, the type of data handled, and the amount of scripting done to dynamically create everything from improted data.

  • However, due to the nature of the API used, more time would've been needed to fully assess the depts of the cyber security surrounding the platform.
    While, different implementations were of course used throughout the development of the project, In order to achieve optimal performance,

    2 functionalities would have to be removed. Or alternatively, if the submission of a node based application were allowed, the web policy's that cause the errors could be resolved.

  • With that being said, I have exhausted as many resources (online/in person) to gain a better understanding of Javascript and web development to even attempt an ambitious project such as this. I think a fair grade would be a B, that taking into account any lack of responsiveness or seperate errors.
